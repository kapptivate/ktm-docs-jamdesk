Execute a campaign and wait for results.

<Warning>
  **Destructive · Writes data · Calls external systems.** This tool can modify or delete data. Review the arguments before letting an agent call it.
</Warning>

## Parameters

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `campaign_json` | string | Yes | Campaign JSON template (from build_campaign) |
| `ignore_errors` | boolean | No | Ignore failures across all action types (options.ignore_failed = &lt;all action codes&gt;). Per-test statuses may be rewritten to success. |
| `max_poll_duration` | string | No | Maximum time to wait for results (default: 10m). Format: Go duration string (e.g. 5m, 15m) |
| `overrides` | string | No | JSON object mapping test IDs to their variable values and owner assignments. Format: \{"&lt;test_id&gt;": \{"input_variables": \{...\}, "direct_owners": \{...\}\}\}. direct_owners maps device identifier to value (e.g., \{"sim_owner": "SIM-001"\}). V3: input_variables values may be either flat strings (legacy) or \{value, bucket\} objects to target a specific variant (variables_v3.md §Bucket System). |
| `parallel` | boolean | No | Run campaign tests in parallel (options.parallel_test_execution=-1). Superadmin-only; the backend rejects test lists that use SIM or SMARTPHONE actions, or sleep on unlocked resources. |

## Description

```text
This is the recommended way to run tests — it blocks until completion and returns per-test results. For very long campaigns (>10 minutes), use start_execution + poll_execution instead.

For ad-hoc runs: build_campaign(test_ids=…) → run_campaign. For a saved Tests campaign: build_campaign(test_campaign_id=…) → run_campaign(overrides=…).
```
