Steps are individual instructions that define what a test does—clicks, taps, text input, waits, and checks. You add steps in the test editor. Each channel supports a different set of step types.

## Web steps

Web tests use browser-based actions to interact with web pages.

**Click** — Press a button, link, or other clickable element. Specify the element by selector (CSS or XPath) or visual marker.

**Type** — Enter text into an input field. Optionally clear the field first and press Enter or Tab after typing.

**Navigate** — Load a new URL. Useful for moving between pages without clicking a link.

**Wait** — Pause the test for a specified duration (in seconds) or until a condition is met (element appears, text is present, etc.).

**Check** — Assert that an element is visible, contains specific text, or meets other conditions. Checks are pass-or-fail; a failed check fails the entire test.

**Scroll** — Move the page up, down, left, or right.

**Hover** — Move the mouse over an element (useful for dropdown menus or tooltips).

**Get property** — Extract and store a value (text, attribute, or element count) for later use.

## Smartphone steps

Smartphone tests interact with mobile apps on Android or iOS devices.

**Tap** — Press a button, link, or UI element. Specify by selector or visual marker.

**Type** — Enter text into a text field. Optionally clear first and press Return or Next after typing.

**Swipe** — Slide across the screen in a direction (up, down, left, right) to scroll lists or dismiss modals.

**Wait** — Pause for a duration or until a UI condition is met (element appears, text is present, etc.).

**Check text** — Assert that a string appears on the current screen. The check fails if the text is not found.

**Check element** — Assert that an element is visible or meets other conditions (enabled, exists, etc.).

**Open app** — Launch the app. Used at the start of tests or to reopen an app after it has been closed.

**Close app** — Exit the current app.

## Common steps (Web and Smartphone)

Both channels support:

**Conditional logic** — Branch the test based on a condition. If a check passes, run one set of steps; if it fails, run another or skip.

**Variable operations** — Set or modify test variables during execution. Use variables to store extracted values or pass data between steps.

**Hooks** — Run setup or teardown logic:
  - **Before Test** — Steps that run once before any test steps (useful for initialization)
  - **Before Step** — Steps that run before each step (useful for logging)
  - **After Step** — Steps that run after each step (useful for recovery)
  - **After Test** — Steps that run once after all test steps (useful for cleanup)

## Step groups

Organize related steps into groups (folders). Groups help keep long tests readable and reusable. You can collapse/expand groups in the editor.

Use groups to:
- Section your test logically (Login, Main Flow, Validation, Cleanup)
- Reuse common workflows across multiple tests (group steps into a template)
- Improve readability in complex tests

## Tips for writing steps

**Use descriptive names** — Name your steps and groups so teammates understand what each does at a glance.

**Break into groups** — Even short tests benefit from grouping. A 20-step test split into 4 groups is easier to debug.

**Add checks frequently** — Checks verify the test is on the right path. A test with no checks might pass but miss real failures.

**Keep it simple** — One step per action. If a step tries to do too much, break it into smaller steps.

**Use variables** — Store dynamic values (IDs, URLs, form data) in variables instead of hardcoding them.
