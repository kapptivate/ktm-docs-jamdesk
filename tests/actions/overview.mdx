An **action** (also called a step) is a single instruction inside a test — click a button, type text, wait for an element, assert a value, send a USSD code. A test is an ordered list of actions executed top-to-bottom on the device assigned to it.

Kapptivate's catalog is organized by **scope** — which kind of test can use the action.

## How actions are scoped

Each action page lists the channels it applies to. The headline rules:

- **Shared** — works in both Web and Smartphone tests. Example: `pause`, `waitfor`, `assertdisplayed`, all the AI verification actions (`aispellcheck`, `aivisualassert`, `aigetmailcode`, `aigetmaillink`, `aigetsmscode`).
- **Web-specific** — only makes sense in browser tests. Example: `hover`, `rightclick`, `switchwindow`, `uploadfile`.
- **Smartphone-specific** — only on a real mobile device. Example: `tap`, `longpress`, `hidekeyboard`.
- **Cellular** — runs on a real SIM, separate from the Web/Smartphone V2 builder. Example: `ussd`, `sms`, `ussd-push`.

A few actions blur the lines — `click` works on both Web and Smartphone (it maps to a tap on touch screens). Pages mention these caveats explicitly in their **Channels** section.

## Step lifecycle

Every action goes through the same lifecycle when it runs:

<Steps>
  <Step title="Resolve">
    The action's selector or input is resolved against the current device (DOM, accessibility tree, screenshot, or cellular session).
  </Step>
  <Step title="Execute">
    The platform issues the action on the device (click, tap, type, USSD send, etc.).
  </Step>
  <Step title="Verify">
    Optional step-level checks run (URL, loading time, step duration) and assertions evaluate the result.
  </Step>
  <Step title="Record">
    Screenshots, timings, and any extracted variables are saved into the execution result.
  </Step>
</Steps>

See [Test builder](/tests/builder) for how steps are added and configured, and [Configurations](/tests/configurations) for variables you can inject into actions.
