An **action** (also called a step) is a single instruction inside a test: click a button, type text, wait for an element, assert a value, send a USSD code. A test is an ordered list of actions executed top-to-bottom on the device assigned to it.

Actions are grouped by **scenario**: which kind of test uses them. Pick a scenario below to see the actions it supports.

## How actions are scoped

- **Shared**: works identically in any test type. Pure infrastructure: `pause`, `waitfor`, `setvariable`, `customstep`, `createmetric`, plus the AI mail/SMS code retrieval actions that query external systems.
- **Web**: interactions and assertions for browser tests. Includes channel-specific actions like `hover`, `rightclick`, `switchwindow`, `uploadfile`, and cross-channel actions like `input`, `scroll`, `assertdisplayed` documented under the Web scope.
- **Smartphone**: touch-based interactions and mobile-specific actions: `tap`, `longpress`, `hidekeyboard`, plus the cross-channel actions documented under the Smartphone scope.
- **SIM (Cellular)**: only inside V1 cellular tests: `ussd`, `sms`, `ussd-push`. See [Cellular tests](/tests/cellular) for the workflow.
- **API**: HTTP requests inside web or mobile tests today. See [API scenarios](/tests/actions/api/overview).

A few actions exist in both Web and Smartphone scopes (e.g., `input`, `scroll`, `assertdisplayed`). They share the same parameters and behavior; each scope has its own page so you can find the action from the scenario you are working on.

## Step lifecycle

Every action goes through the same lifecycle when it runs:

<Steps>
  <Step title="Resolve">
    The action's selector or input is resolved against the current device: DOM, accessibility tree, screenshot, or cellular session.
  </Step>
  <Step title="Execute">
    The platform issues the action on the device (click, tap, type, USSD send, etc.).
  </Step>
  <Step title="Verify">
    Optional step-level checks run (URL, loading time, step duration) and assertions evaluate the result.
  </Step>
  <Step title="Record">
    Screenshots, timings, and any extracted variables are saved into the execution result.
  </Step>
</Steps>

See [Test builder](/tests/builder) for how steps are added and configured, and [Configurations](/tests/configurations) for variables you can inject into actions.
