Assertion and extraction actions let you verify the state of your app (an element is visible, a value matches expectations) and pull data out for use in subsequent steps. Unlike interaction actions, these don't change the app state—they observe and report.

## Assertions

### assertdisplayed

Assert that an element is visible and present on the page.

| Parameter | Type | Required | Notes |
|-----------|------|----------|-------|
| `locator` | string | Yes | CSS/XPath selector for the element. |
| `timeout` | number (ms) | No | Max time to wait for the element. Default ~5–10 seconds. |
| `wait_for` | object | No | Advanced wait condition (clickable, displayed, enabled, exist). |

**Applies to**: Web, Smartphone

**Step-level checks available**: `url`, `loading_time`, `step_duration`

The step fails if the element is not found or not visible within the timeout. Use `wait_for: { clickable: { timeout: 5000 } }` to assert the element is not just visible but also clickable.

### assertnotdisplayed

Assert that an element is NOT visible or does not exist.

| Parameter | Type | Required | Notes |
|-----------|------|----------|-------|
| `locator` | string | Yes | CSS/XPath selector for the element. |
| `timeout` | number (ms) | No | Max time to verify absence. |
| `wait_for` | object | No | Advanced wait condition. |

**Applies to**: Web, Smartphone

The step fails if the element is found and visible. Use this to verify a loading spinner has disappeared, an error message is gone, or a modal has closed.

## Data extraction

### gettext

Extract the text content of an element.

| Parameter | Type | Required | Notes |
|-----------|------|----------|-------|
| `locator` | string | Yes | CSS/XPath selector for the element. |
| `variable` | string | No | Test variable name to store the text. If omitted, text is logged but not stored. |
| `wait_for` | object | No | Pre-extraction wait condition. |
| `timeout` | number (ms) | No | Max time to find the element. |

**Applies to**: Web, Smartphone

**Step-level checks available**: `step_duration`

Reads the `.textContent` property of the element. Use this to capture dynamic text like a confirmation message, order number, or user greeting.

### getvalue

Extract the value of a form input or input-like element.

| Parameter | Type | Required | Notes |
|-----------|------|----------|-------|
| `locator` | string | Yes | CSS/XPath selector for the input/select/textarea. |
| `variable` | string | No | Test variable name to store the value. |
| `wait_for` | object | No | Pre-extraction wait condition. |
| `timeout` | number (ms) | No | Max time to find the element. |

**Applies to**: Web, Smartphone

**Step-level checks available**: `step_duration`

Reads the `.value` property of form elements. Use this to verify a form was pre-filled, or to capture user-entered data for downstream assertions.

### gettextorvalue

Extract text or value (fallback extraction).

| Parameter | Type | Required | Notes |
|-----------|------|----------|-------|
| `locator` | string | Yes | CSS/XPath selector. |
| `variable` | string | No | Test variable name to store the result. |
| `wait_for` | object | No | Pre-extraction wait condition. |
| `timeout` | number (ms) | No | Max time to find the element. |

**Applies to**: Web, Smartphone

**Step-level checks available**: `step_duration`

Attempts to read `.textContent` first; if empty, falls back to `.value`. Useful when you're not sure whether an element contains text or a form value.

### find

Locate an element without performing any action.

| Parameter | Type | Required | Notes |
|-----------|------|----------|-------|
| `locator` | string | Yes | CSS/XPath selector. |
| `wait_for` | object | No | Condition to wait for (clickable, displayed, enabled, exist). |
| `timeout` | number (ms) | No | Max time to find the element. |

**Applies to**: Web, Smartphone

**Step-level checks available**: `step_duration`

This is a pure locator step—it finds an element and validates its state but doesn't click, type, or interact. Use it to verify an element exists before taking action, or as part of the test setup to "warm up" element caches.

## Channel-specific extraction

For channel-specific data extraction (such as extracting structured data from USSD responses or cellular-specific content), refer to [Cellular testing](/tests/cellular) for SMS and USSD-specific actions. The `extract` action is a general-purpose step available for custom extraction logic and is typically configured at the test or channel level.

## Step-level checks

You can add checks to any action—assertion, extraction, interaction, or navigation—to verify properties like the URL, page load time, or step duration.

### url

Verify the current page URL matches an expected value.

| Property | Type | Values |
|----------|------|--------|
| `entity` | string | `"url"` |
| `operator` | string | `equals`, `contains`, `matches` (regex), `starts_with`, `ends_with` |
| `value` | string (templated) | The expected URL or pattern. |
| `statusModifier` | string | `"error"` (default, fails step) or `"warning"` (logs but continues) |

**Example**:
```json
{
  "entity": "url",
  "operator": "contains",
  "value": "/dashboard",
  "statusModifier": "error"
}
```

### loading_time

Verify the page load time is within bounds.

| Property | Type | Values |
|----------|------|--------|
| `entity` | string | `"loading_time"` |
| `operator` | string | `less_than`, `less_than_or_equal`, `greater_than`, `equal`, `not_equal` |
| `value` | number | Time in milliseconds. |
| `statusModifier` | string | `"error"` (default) or `"warning"` |

**Example**:
```json
{
  "entity": "loading_time",
  "operator": "less_than",
  "value": "3000",
  "statusModifier": "warning"
}
```

### step_duration

Verify a single step's execution time is within bounds.

| Property | Type | Values |
|----------|------|--------|
| `entity` | string | `"step_duration"` |
| `operator` | string | `less_than`, `less_than_or_equal`, `greater_than`, `equal`, `not_equal` |
| `value` | number | Time in milliseconds. |
| `statusModifier` | string | `"error"` (default) or `"warning"` |

**Example**:
```json
{
  "entity": "step_duration",
  "operator": "less_than",
  "value": "5000"
}
```

## Composing checks with logical operators

You can combine multiple checks with `and` or `or` logic:

```json
{
  "type": "group",
  "logicalOperator": "and",
  "checks": [
    { "entity": "url", "operator": "contains", "value": "/success" },
    { "entity": "loading_time", "operator": "less_than", "value": "2000" }
  ]
}
```

This example passes only if the URL contains "/success" AND the page loaded in under 2 seconds.
