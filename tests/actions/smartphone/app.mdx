Launch a new mobile app or switch to an already-running app during test execution. Unlike [Starting App](/tests/actions/smartphone/startingapp), this action can appear anywhere in your test, allowing you to test multi-app workflows.

## Channels

- ❌ Web
- ✅ Smartphone (Android/iOS)
- ❌ Cellular

App launch is smartphone-only.

## Parameters

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| app | Template (string) | Yes | App identifier: Android package ID or iOS bundle ID. Supports templating (e.g., `${APP_PACKAGE}`). |
| meta | JSON object | No | Application metadata (name, label, package, version, display_name). Read-only; populated from app registry. |

## Example

</Tab>

## Notes

- App is a runtime action and can appear anywhere after Starting App in your test.
- If the app is not installed on the device, the test will fail at execution.
- Switching to an already-running app brings it to the foreground without restarting it (if the app supports background activity).
- App identifiers support [templating](/concepts/core-concepts) for dynamic app selection across environments.
- Metadata (name, label, version) is automatically populated when the app is selected; it describes the application in test reports.
- Step-level checks (step_duration) are available after app launch/switch completes.
