Set the starting app for your smartphone test. This action must be the first step in any smartphone test and specifies the initial app context. The app is identified by its package ID (Android) or bundle ID (iOS).

## Channels

- ❌ Web
- ✅ Smartphone (Android/iOS)
- ❌ Cellular

Starting App is smartphone-only and acts as the test entry point.

## Parameters

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| app | Template (string) | Yes | App identifier: Android package ID or iOS bundle ID. Supports templating (e.g., `${APP_PACKAGE}`). |
| meta | JSON object | No | Application metadata (name, label, package, version, display_name). Read-only; populated from app registry. |

## Example

<Tabs>
<Tab label="Android">

```json
{
  "type": "startingapp",
  "version": "v1.0.0",
  "inputs": {
    "app": "com.android.chrome"
  }
}
```

</Tab>
<Tab label="iOS">

```json
{
  "type": "startingapp",
  "version": "v1.0.0",
  "inputs": {
    "app": "com.apple.mobilesafari"
  }
}
```

</Tab>
</Tabs>

## Notes

- Starting App is always Step 0 (index 0) in smartphone tests and cannot be moved.
- The app package ID (Android) or bundle ID (iOS) is resolved from the app registry on your device.
- Metadata (name, label, version) is automatically populated when the app is selected; it describes the application in test reports.
- If the app is not installed on the device, the test will fail at execution.
- App identifiers support [templating](/concepts/core-concepts) for dynamic environments (e.g., staging vs. production apps).
- Step-level checks (step_duration) are available after app launch completes.
