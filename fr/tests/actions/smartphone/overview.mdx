Les tests smartphone exécutent votre application native sur de vrais appareils Android et iOS. Ils sont construits à partir des steps ci-dessous ; les groupes correspondent aux catégories de la palette **Ajouter un step** dans le [test builder](/fr/tests/builder), ce que vous voyez ici est ce que vous voyez dans le produit.

Chaque test smartphone commence par un step verrouillé **Launch app** qui ouvre l'application sélectionnée lors de la création du test.

## Interactions

| Step | Description |
|------|-------------|
| [Tap](/fr/tests/actions/smartphone/tap) | Appuyer sur un élément, ou sur une position X/Y exacte en mode Coordinates |
| [Long press](/fr/tests/actions/smartphone/longpress) | Appui long pour ouvrir les actions contextuelles |
| [Hide keyboard](/fr/tests/actions/smartphone/hidekeyboard) | Masquer le clavier virtuel |
| [Fill input](/fr/tests/actions/smartphone/input) | Saisir du texte, des variables ou des données générées dans un champ |

## Email

| [Get email link](/fr/tests/actions/smartphone/getemaillink) | Attendre un email et extraire un lien dans une variable |

## Vérifications

| Step | Description |
|------|-------------|

## Paramètres communs des steps

Chaque step partage quelques paramètres, visibles dans le panneau latéral du step dans le builder :

- **Ignorer l'erreur sur ce step** (`ignore_errors`) : l'exécution continue avec le step suivant même si celui-ci échoue.
- **Désactiver pendant l'exécution** (`disabled`) : le step reste dans le scénario mais n'est pas exécuté.
- **Capabilities personnalisées** (`use_capabilities`, `capabilities`) : ajuster ou remplacer le comportement d'exécution pour ce step uniquement.

Les steps qui ciblent un élément partagent aussi les paramètres IA : une **description** en langage naturel, un **selector manuel** optionnel (XPath dans l'arbre UI de l'application), et une **stratégie IA** qui détermine quand l'IA localise l'élément par rapport au selector enregistré. Sur les steps d'interaction smartphone, l'IA est désactivée par défaut ; activez-la par step ou via la stratégie du test. Voir [Selectors IA](/fr/ai/selectors).

## Des résultats aux analytiques

Chaque step enregistre son timing, son statut et ses captures d'écran, ainsi que les variables créées. Au niveau du test, chaque exécution alimente la mesure `test` utilisée par les [Analytiques](/fr/monitoring/dashboards) : durée, compteurs de succès/avertissement/échec, et en cas d'échec, quel type de step a provoqué l'erreur. Les métriques personnalisées créées dans les steps remontent au même endroit.
