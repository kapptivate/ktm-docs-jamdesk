Une **action** (ou step) est une instruction unique dans un test : cliquer sur un bouton, saisir du texte, vérifier un élément, envoyer un code USSD. Un test est une liste ordonnée d'actions exécutées de haut en bas sur l'appareil qui lui est assigné.

La bibliothèque est organisée par **canal**, et à l'intérieur de chaque canal elle reflète la palette de steps que vous voyez dans le produit. Un step qui existe sur deux canaux possède une page par canal, vous le trouvez donc toujours là où vous travaillez.

## Ce que chaque page de step vous indique

Chaque page suit la même structure, ce qui vous permet de parcourir la bibliothèque comme une référence :

- **Paramètres** : chaque champ tel qu'il apparaît dans le builder, avec son nom JSON, sa valeur par défaut et son comportement.
- **Dans le scénario JSON** : un objet step complet, prêt à être collé dans un scénario.
- **Résultats et métriques** : ce que le step enregistre lors de son exécution (temps, statut, captures d'écran, variables) et ce qui remonte vers Analytics.

Vous pouvez également lister les types d'actions disponibles de manière programmatique : exécutez [ktm tests action-types](/fr/cli/commands/tests/action-types) depuis le CLI, ou appelez [list_action_types](/fr/mcp/tools/test-management/list_action_types) depuis le serveur MCP.
