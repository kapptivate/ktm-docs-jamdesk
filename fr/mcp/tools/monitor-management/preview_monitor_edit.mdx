Prévisualiser l'impact de la modification d'un monitor SANS appliquer les changements.

<Note>
  **Lecture seule · Appelle des systèmes externes.** Appel sûr : cet outil ne modifie pas les données.
</Note>

## Paramètres

| Champ | Type | Requis | Description |
| --- | --- | --- | --- |
| `active` | boolean | Non | Nouveau statut actif |
| `input_variables` | string | Non | Nouveau JSON d'input variables. Les buckets sont injectés automatiquement depuis le tableau variants lors de la soumission. L'aperçu montre le résultat projeté. |
| `monitor_id` | number | Oui | Identifiant du monitor pour lequel prévisualiser les changements |
| `name` | string | Non | Nouveau nom du monitor |
| `owners` | string | Non | Nouveau JSON d'owners |
| `scheduling` | string | Non | Nouveau JSON de planification |
| `variants` | array | Non | Épingler les variants pour les groupes de variables du monitor. Chaque entrée est "5" (ID numérique du variant) ou "GroupName:VariantName". Les groupes non épinglés conservent leur bucket existant (préservation en mode édition). Utilisez ceci pour ré-épingler des groupes spécifiques pendant l'édition. |

## Exemple d'arguments

Arguments illustratifs qu'un agent fournit lors de l'appel de cet outil :

```json
{
  "monitor_id": 0
}
```

## Description

```text
Renvoie un diff côté client montrant les valeurs actuelles vs proposées pour chaque champ modifié, plus l'aperçu de résolution de bucket lorsque des variants sont fournis ou que l'auto-migration se déclencherait.

Épinglez les variants via le tableau variants (même format que create_monitor). Lorsque le monitor est legacy (variable_group != 0 ou input_variables sans buckets), cet aperçu montre ce que l'auto-migration ferait, utile pour les utilisateurs qui veulent inspecter le plan de migration avant d'appeler edit_monitor.

Workflow : get_monitor → preview_monitor_edit → (examiner le diff + plan de buckets) → edit_monitor.
```
